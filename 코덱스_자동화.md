# 코덱스 자동화

- 원문: OpenAI Developers - Automations  
  https://developers.openai.com/codex/app/automations
- 확인일: 2026년 6월 9일
- 용도: 2시간 Codex 실습 강의 및 회사 업무 자동화 아이디어 정리

## 한 줄 요약

Codex 자동화는 반복해서 확인해야 하는 작업을 정해진 시간마다 백그라운드에서 실행하게 만드는 기능입니다. 결과가 있으면 Codex 앱의 Triage/inbox에 남기고, 보고할 내용이 없으면 자동으로 정리됩니다.

## 자동화로 할 수 있는 일

- 매일 아침 프로젝트 변경사항 요약 받기
- 장시간 실행 중인 명령이나 배포 상태 확인하기
- GitHub PR 상태, 리뷰 코멘트, CI 실패 여부 확인하기
- 최근 커밋에서 생긴 버그 후보를 찾아 수정하기
- 자주 반복하는 업무를 Skill과 연결해 실행하기
- 특정 프로젝트나 여러 프로젝트를 정기적으로 점검하기

## 자동화 실행 조건

프로젝트 범위 자동화를 쓰려면 다음 조건이 필요합니다.

- 로컬 Codex 앱이 실행 중이어야 합니다.
- 자동화를 실행할 컴퓨터가 켜져 있어야 합니다.
- 선택한 프로젝트 폴더가 디스크에 그대로 있어야 합니다.

Git 저장소에서는 자동화를 로컬 프로젝트에서 직접 실행할지, 새 worktree에서 분리해서 실행할지 선택할 수 있습니다. Git 저장소가 아닌 프로젝트는 해당 프로젝트 디렉터리에서 바로 실행됩니다.

## 자동화 결과 관리

Codex 앱 사이드바의 Automations 패널에서 자동화와 실행 기록을 볼 수 있습니다.

Triage 영역은 자동화 결과를 모아두는 inbox 역할을 합니다. 결과가 있는 실행은 Triage에 표시되고, 전체 실행 또는 읽지 않은 실행만 필터링해서 볼 수 있습니다.

## 자동화 유형

### Standalone automation

각 실행이 새로 시작되는 독립 작업입니다.

이런 경우에 적합합니다.

- 매번 새 보고서를 만들 때
- 하루 또는 주 단위로 프로젝트 상태를 점검할 때
- 하나의 자동화를 여러 프로젝트에 적용할 때
- 결과를 Triage에 별도 실행 기록으로 남기고 싶을 때

사용자 지정 주기가 필요하면 cron 문법으로 스케줄을 지정할 수 있습니다.

### Project automation

특정 프로젝트를 대상으로 반복 실행되는 자동화입니다.

Git 저장소에서는 다음 중 하나를 고릅니다.

- Local mode: 현재 체크아웃에서 직접 실행합니다.
- Worktree mode: 별도 worktree를 만들어 실행합니다.

미완성 작업과 자동화 변경사항을 분리하고 싶다면 worktree가 안전합니다. 반대로 현재 작업 폴더를 바로 점검해야 한다면 local mode를 쓸 수 있지만, 자동화가 사용자가 편집 중인 파일을 건드릴 수 있다는 점을 고려해야 합니다.

### Thread automation

현재 대화 스레드에 붙어 있는 반복 알림입니다. Codex가 같은 대화로 다시 돌아와야 할 때 사용합니다.

이런 경우에 적합합니다.

- 긴 명령이 끝났는지 계속 확인하기
- 배포가 끝날 때까지 같은 스레드에서 추적하기
- Slack, GitHub 등 연결된 소스를 주기적으로 확인하기
- 리뷰 루프를 일정 간격으로 이어가기
- 조사, 장애 triage처럼 맥락이 이어져야 하는 작업 유지하기

Thread automation 프롬프트는 오래 버틸 수 있게 써야 합니다. 매번 깨어날 때 무엇을 해야 하는지, 보고할 내용이 있는지 어떻게 판단할지, 언제 멈추거나 사람에게 물어봐야 하는지를 포함하는 것이 좋습니다.

## Codex에게 자동화를 만들어 달라고 하기

일반 Codex 스레드에서 자연어로 자동화를 만들거나 수정해 달라고 요청할 수 있습니다.

요청할 때 포함하면 좋은 정보는 다음과 같습니다.

- 어떤 작업을 반복할지
- 언제 또는 얼마나 자주 실행할지
- 현재 스레드에 붙일지, 새 실행으로 시작할지
- 어떤 프로젝트를 대상으로 할지
- 결과가 있을 때 어떻게 보고할지
- 아무 변화가 없을 때 어떻게 처리할지

예시 요청:

```text
매주 월요일 오전 9시에 이 프로젝트의 지난 7일 변경사항을 요약해서 Triage에 보고하는 자동화를 만들어줘.
```

```text
이 스레드에서 10분마다 배포 로그를 확인하고, 성공하면 요약하고 멈춰줘. 실패하면 원인 후보와 다음 조치를 알려줘.
```

## Skill과 함께 쓰기

자동화는 Codex가 사용할 수 있는 플러그인과 Skill을 함께 사용할 수 있습니다.

반복 업무를 안정적으로 관리하려면 자동화 프롬프트 안에 모든 절차를 길게 쓰기보다 Skill로 절차를 정의해두고 자동화에서 `$skill-name`처럼 명시적으로 호출하는 방식이 좋습니다.

예시:

```text
매일 오전 10시에 GitHub PR 상태를 확인하고 새 리뷰 코멘트가 있으면 $pr-review-fix skill을 실행해줘.
```

## 자동화 테스트 방법

자동화를 예약하기 전에 일반 Codex 스레드에서 같은 프롬프트를 먼저 실행해보는 것이 좋습니다.

확인할 점은 다음과 같습니다.

- 프롬프트 범위가 명확한가?
- 모델, reasoning effort, 도구 선택이 기대대로 동작하는가?
- 결과 diff가 리뷰 가능한 수준인가?
- 보고할 내용과 조용히 넘어갈 조건이 명확한가?

처음 몇 번의 예약 실행 결과를 보고 프롬프트와 실행 주기를 조정하면 안정성이 좋아집니다.

## Worktree 정리

Git 저장소에서 worktree 모드 자동화를 자주 실행하면 시간이 지나면서 worktree가 많이 쌓일 수 있습니다.

필요 없는 자동화 실행 기록은 archive하고, 계속 보존할 필요가 있는 실행만 pin하는 습관이 좋습니다.

## 권한과 보안

자동화는 사용자가 지켜보지 않는 상태에서 실행되므로 기본 sandbox 설정의 영향을 받습니다.

### Read-only

파일 수정, 네트워크 접근, 컴퓨터 앱 조작이 필요한 작업은 실패할 수 있습니다.

### Workspace-write

워크스페이스 안의 파일 작업은 가능하지만, 워크스페이스 밖 파일 수정이나 네트워크 접근 등은 제한될 수 있습니다. 필요한 명령만 rules로 허용하는 방식이 좋습니다.

### Full access

Codex가 백그라운드에서 파일을 바꾸고 명령을 실행하고 네트워크에 접근할 수 있으므로 위험이 커집니다. 회사 업무에서는 가능한 workspace-write를 기본으로 두고, 꼭 필요한 명령만 허용하는 쪽이 안전합니다.

관리형 환경에서는 관리자가 승인 정책이나 sandbox 모드를 제한할 수 있습니다. 조직 정책이 `approval_policy = "never"`를 허용하지 않으면 자동화도 선택한 모드의 승인 동작을 따릅니다.

## 회사 업무용 자동화 예시

### 1. 매일 아침 프로젝트 브리핑

```text
매일 오전 9시에 origin/main 기준으로 지난 24시간 동안 이 프로젝트에서 바뀐 내용을 업무 흐름별로 묶어 요약해줘. 커밋 해시 나열보다 무엇이 왜 바뀌었는지 중심으로 설명해줘.
```

### 2. PR 리뷰 코멘트 추적

```text
30분마다 현재 PR의 새 리뷰 코멘트와 CI 상태를 확인해줘. 새로 대응할 항목이 있으면 작업 목록으로 정리하고, 없으면 조용히 넘어가줘.
```

### 3. 배포 감시

```text
이 스레드에서 10분마다 배포 상태를 확인해줘. 성공하면 배포 요약을 남기고 멈춰줘. 실패하면 실패 로그, 원인 후보, 롤백 필요 여부를 정리해줘.
```

### 4. 최근 커밋 버그 점검

```text
매일 오후 5시에 내 최근 24시간 커밋을 확인하고, 변경 파일 중심으로 작은 테스트나 린트를 실행해 버그 후보를 찾아줘. 실제 문제가 확인되면 최소 수정안을 제안해줘.
```

### 5. 교육 준비 자동화

```text
매일 오전 8시에 Codex 실습 강의 준비 폴더를 확인하고 새로 추가된 문서, 사진, Markdown 파일을 요약해줘. 강의안에 반영할 만한 항목이 있으면 Triage에 보고해줘.
```

## 실습 강의에서 강조할 말

자동화는 "Codex에게 한 번 시키는 일"을 "Codex가 주기적으로 챙기는 일"로 바꾸는 기능입니다. 다만 백그라운드에서 혼자 실행되기 때문에, 프롬프트에는 작업 범위, 보고 기준, 멈출 조건, 보안 제한을 분명히 넣어야 합니다.
